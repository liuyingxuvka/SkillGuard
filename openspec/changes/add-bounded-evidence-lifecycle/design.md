## Context

SkillGuard's current owner runner stores complete stdout/stderr as raw content-addressed `.bin` files and treats their presence as part of immutable receipt validity. Content addressing prevents byte-for-byte duplicates, but the store has no declared roots, retention states, collection plan, quarantine, or purge boundary. A FlowGuard self-check also emits very large successful model reports, so four outputs alone occupy about 954 MiB. The same maintenance unit currently exposes more than one physical evidence root, while installed smoke checks generate avoidable Python bytecode.

The existing contract already separates execution owners from semantic check projections, but the compiler's implicit signature grouping and model-assertion defaults do not express the intended authority clearly. Sharing must be authored explicitly by the target/compiler contract, never guessed from equivalent command text.

## Goals / Non-Goals

**Goals:**

- Preserve every target-declared semantic check, obligation, projection, and evidence-domain identity while executing an explicitly shared producer at most once.
- Store complete streams losslessly and deterministically with materially lower disk use.
- Make every retained evidence object explainable through a root and transitive reference path.
- Make audit and planning provably read-only; make collection quarantine-first and purge separately authorized.
- Keep current/release-pinned replay valid through compaction and cleanup.
- Keep installation projection limited to consumer-runtime content and leave no generated bytecode in an installed stage.

**Non-Goals:**

- SkillGuard will not decide whether a target skill's declared capability is sufficiently deep.
- SkillGuard will not invent target checks, obligations, semantic categories, evidence domains, or producer-sharing declarations.
- The evidence lifecycle will not be copied into graduated consumer skills or used by their ordinary domain runtime.
- The normal runtime will not read old uncompressed sidecars through a compatibility or fallback path.
- Retention will not guess importance from age, filename, command similarity, or output size.

## Decisions

### 1. Keep semantic projections separate from producer execution

Each declared check remains a semantic projection with its own `check_id`, `semantic_check_id`, covered obligations, evidence class/domain, dependency declarations, and projection hash. A producer is the execution owner row. The compiler assigns a private unique producer to a check by default; several checks share one producer only when their current source contract explicitly declares the same execution owner and their producer behavior, inputs, target-input roles, toolchain, and environment agree exactly. Their external owner dependencies are combined after the semantic projections are mapped; evidence domains and check dependencies do not become producer identity.

Command/argument/name equality is not a sharing rule. The prior signature-based implicit grouping is removed. This keeps the existing owner/receipt machinery while making the authority source explicit and target-owned.

### 2. Split logical evidence identity from physical storage identity

Stream references record the SHA-256 and length of the original bytes plus the SHA-256 and length of a deterministic gzip representation (`mtime=0`). The logical hash remains the execution-result identity; the storage hash addresses the physical object. Complete streams are written incrementally to a temporary file and atomically published into one maintenance-unit store.

Replay first verifies path containment, stored length/hash and encoding, then safely decompresses while enforcing the declared logical length and configured safety ceiling, and finally verifies the logical hash. Small JSON result and termination sidecars retain the existing content-reference shape.

### 3. One canonical evidence store per maintenance unit

The frozen maintenance-unit plan declares exactly one `owner_evidence_root`. Heads, receipts, blobs, lifecycle pins, quarantine journals, and purge receipts resolve beneath that root. A second writable authority is a planning blocker. Run-local folders may contain temporary capture files only until atomic publication finishes; they are not receipt authority.

### 4. Reachability, not time, controls retention

Lifecycle roots include one mutable current-head authority per maintenance-unit producer, one mutable current-aggregation authority per maintenance-unit member/profile, active/running attempts, current closure receipts, installation and release pins, retained failed diagnostics, and explicitly pinned historical receipts. Each authority references one exact immutable object; historical head and aggregation files are not current merely because they remain in their content-addressed directories. TestMesh installation and global-prompt bindings are typed external-domain identities verified by aggregation replay, not objects owned or traversed by the owner-evidence collector. Transitive internal references classify objects as `current`, `pinned`, `historical_referenced`, `failed_diagnostic`, `quarantined_corrupt`, `orphan`, or `temporary`.

The first implementation is intentionally conservative: ambiguity, corruption, path escape, a missing referenced object, a reference cycle outside the allowed schema, multiple stores, a missing current-head authority, or an active writer blocks mutation. Writers create an active marker under a short store-wide lifecycle barrier, run without holding the barrier, publish the new head and current authority, then remove their marker under the same barrier. Apply and purge hold that barrier across fresh audit and mutation, preventing a writer/collector race. Age can satisfy a quarantine grace period but cannot make a reachable object collectible.

### 5. Four lifecycle commands with strict mutation boundaries

- `evidence-audit` reads the store and emits classifications/findings to stdout without filesystem writes.
- `evidence-gc-plan` reads the same snapshot and emits a canonical plan with inventory hash and plan hash; it writes nothing.
- `evidence-gc-apply` accepts an explicit plan, revalidates the frozen snapshot, and atomically moves only exact unreachable candidates into an explicit quarantine root, recording a journal/receipt.
- `evidence-gc-purge` accepts the exact apply receipt and plan hash, rechecks current reachability and grace policy, then permanently removes only the named quarantine objects and records hashes/byte counts.

The operation id is deterministic, and one durable mutable journal records the pre-mutation and post-mutation state of every item. Atomic publication of that journal uses bounded retry for transient Windows permission/sharing holds. If a process stops after deletion but before the post-mutation journal write, the next invocation must resume the same operation id, recognize the recorded `prepared` state, and complete it without inventing another cleanup owner.

Normal validation may remove only its own unpublished temporary capture files. It cannot invoke purge or silently collect persistent evidence.

### 6. Direct current replacement

The `.current` receipt schema moves directly to compressed stream references. Old uncompressed evidence remains outside current authority and is eligible only for conservative archive/quarantine planning. There is no ordinary reader, converter, migration command, alias, or dual write.

### 7. Installation projection is derived from installable components only

Compiled/check-manifest identities used by installed parity are projected from `projection:installation`; source-only tests, workflows, models, reports, and author-maintenance fingerprints are excluded. Installed smoke launches Python with bytecode disabled. A clean installed stage must contain no `.pyc` or `__pycache__` residue.

## Risks / Trade-offs

- **A current receipt becomes unreadable after direct replacement** → freeze the new store identity, treat old evidence as archive-only, run one new final validation, and pin the new release receipt before any purge.
- **Compression or decompression consumes excessive memory** → use bounded streaming I/O and enforce both compressed and logical byte budgets.
- **A stale plan moves newly reachable evidence** → bind plans to the exact root identity and inventory snapshot; any drift produces zero moves.
- **Quarantine partially completes** → journal each atomic move and make repeated apply idempotent and recoverable; purge rejects incomplete journals.
- **Explicit sharing is underdeclared** → execute separate producers; correctness is preferred over automatic deduplication.
- **Explicit sharing is declared incorrectly** → fail before execution when producer behavior, input selectors, target-input roles, toolchain, or environment disagree. Different evidence domains and semantic dependency rows remain visible in the separate check projections.

## Migration Plan

1. Add schema/model/tests for producer projections and compressed evidence before changing the current writer.
2. Directly replace current writer/replay paths and lifecycle CLI; run focused tests against temporary stores.
3. Recompile the SkillGuard self contract and update the FlowGuard validation-composition model.
4. Freeze and execute exactly one current full SkillGuard validation, then install transactionally with bytecode-disabled smoke.
5. Pin the release receipt and run lifecycle audit/plan against the old working evidence. Quarantine only objects proven unreachable under the new roots.
6. Replay current and release-pinned receipts. Purge quarantine only after the explicit grace/replay gates pass.
7. Publish the new source-only SkillGuard release. Rollback before purge consists of restoring the previous installed projection and leaving quarantined objects intact; after purge, recovery depends on Git/release artifacts and any explicitly retained external backup.
