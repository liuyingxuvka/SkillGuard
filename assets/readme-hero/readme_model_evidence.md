# SkillGuard README Model Evidence

This file records the LogicGuard-backed capability model used for the `v0.3.4`
source README. It is README synthesis evidence only. It does not replace
SkillGuard or FlowGuard checks, package publication, GitHub release
confirmation, or future AI behavior validation.

## Repository Fact Ledger

- product surface: SkillGuard is an author-side Codex skill maintenance,
  execution-depth, graduation, and clean consumer-distribution system.
- entry points: README, `.agents/skills/skillguard/SKILL.md`, dispatcher,
  compiler, author supervisor, TestMesh, Portfolio, consumer distribution,
  installer, privacy/provenance checks, FlowGuard models, schemas, fixtures,
  tests, and CI workflow definition.
- release/version facts: current source version is `v0.3.4`; `VERSION`,
  `pyproject.toml`, README, changelog, and release notes must agree before a tag
  or GitHub Release claim.
- authority facts: the `.skillguard` contract trio is author-maintenance
  authority only; graduated consumer skills exclude it and work independently.
- provider facts: official OpenSpec is external and unmanaged; FlowGuard reads
  only proposal, design, specs, tasks, and status context.
- privacy-sensitive exclusions: public and consumer material excludes local
  absolute paths, private skill inventories, task text, credentials, receipts,
  run state, Portfolio state, and author router state.

## LogicGuard-Backed Capability Model

### Root Claim

SkillGuard helps an explicit maintainer make each skill's promises testable,
prove completion and depth with that skill's own checks, and graduate a clean
standalone consumer without making SkillGuard a runtime dependency.

### Mechanism

SkillGuard binds one maintenance unit, its members, semantic checks, evidence
subjects, owners, inputs, dependencies, toolchain, and environment. It compiles
author authority, freezes the exact check inventory, executes or same-unit
single-flights exact owners, records immutable terminal evidence, and blocks
closure on missing, duplicate, failed, skipped, stale, foreign-unit,
non-terminal, or cleanup-unconfirmed evidence. It then builds and audits a
consumer projection that excludes every SkillGuard control surface.

### Evidence

- current author schema, compiler, supervisor, receipt, TestMesh, Portfolio,
  consumer distribution, installation, privacy, and project-adoption source;
- `.flowguard` declared-check, validation-composition, executable-contract,
  runtime-authority, Portfolio, template, router, and process models;
- positive and known-bad model scenarios for author boundaries, unit evidence
  isolation, ordinary-project zero writes, and consumer cleanliness;
- focused tests for exact execution identity, foreign-unit rejection,
  same-unit concurrency, target-owned runtime preflight, staged install,
  rollback, and OpenSpec exclusion;
- public README, hero prompt/design evidence, version files, changelog, and
  current command surface.

### Reader Value

Readers can distinguish author contract mapping, actual execution depth,
maintenance-unit graduation, clean consumer installation, and publication.
These are related but separate claims.

### Boundary

The README may claim author-side current contracts, exact declared-check
reconciliation, same-unit evidence, independent Portfolio status, clean
consumer projection, ordinary-project zero writes, private author routing,
staged install/rollback, and external OpenSpec exclusion only where current
checks support them. It must not claim packaged CLI distribution, hosted
service behavior, remote CI success, package publication, domain truth, or
future AI correctness without separate evidence.

### Objection And Rebuttal

Objection: removing contracts from consumers might weaken quality.

Rebuttal: quality evidence remains mandatory before graduation, but it stays
with the maintainer. The consumer retains its own entrypoint, runtime, native
checks, and bounded claims; it no longer depends on SkillGuard metadata.

Objection: independent units may repeat similar commands.

Rebuttal: command similarity is not semantic ownership. If two maintained
skills truly promise the same behavior, the boundary must be split, merged, or
retired. Otherwise each unit owns and proves its own distinct promise.

## Narrative Structure Plan

- first-screen promise: train and verify on the maintainer computer; ship an
  independent consumer.
- section order: why the boundary matters, contract versus execution depth,
  capability table, status, exact command surface, author runtime, consumer
  projection, native ownership, workflows, validation, non-goals, public
  boundary, layout, history, license.
- visual proof placement: the hero directly under the H1 shows author
  maintenance separating from consumer runtime.
- quick-start placement: command examples follow the boundary and capability
  explanation.
- limitation placement: validation, non-goals, and public boundary appear
  before repository layout and release history.
- public/private boundary placement: the maintainer-only authority boundary is
  stated before any installation or release claim.

## Gap Ledger

- unsupported claims: packaged CLI distribution, hosted service, binary
  release, remote CI success, package publication, universal domain
  correctness, and future AI behavior.
- missing evidence: external services and public release publication remain
  outside this source-only evidence.
- migration risk: target-domain runtime under `.skillguard/runtime` must move
  and pass parity before cleanup; unknown canonical sources block destructive
  withdrawal.
- maturity: SkillGuard remains a `0.3.x` source-level project.
- privacy risks: public and consumer artifacts must exclude private installed
  inventories, absolute paths, credentials, author receipts, and task text.

## README Synthesis Use

The README leads with the author/consumer separation, then explains why exact
declared checks and independent maintenance-unit evidence matter. It presents
the current public command surface, private author routing, clean consumer
distribution, external OpenSpec boundary, validation limits, and source-only
release status without converting any local check into a publication claim.
